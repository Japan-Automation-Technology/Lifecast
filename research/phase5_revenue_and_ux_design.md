# Phase 5 設計検討: 収益モデル × UX × Apple IAP戦略

**検討日: 2026-02-14**

---

## 核心的な設計判断: 「投げ銭」を廃止し、全てを「応援」に統一する

### 背景

従来の設計では、2つの取引類型を想定していた:
- **クラウドファンディング支援**（プロジェクト単位、リターンあり）
- **投げ銭**（動画単位、リターンなし）

しかし検討の結果、この区別はユーザーにとって不要であり、プラットフォームにとっても不利益であることが判明した。

### 問題点

1. **UXの複雑化**: ユーザーは「これは投げ銭？支援？」と考える必要がない。動画を見て「いいな」と思って金を出す行為は1つ
2. **収益の取りこぼし**: 投げ銭を100%クリエイター還元（IAP回避のため）にすると、PF手数料を取れない。全取引の40%前後が無収益になりうる
3. **Apple審査リスク**: 投げ銭機能があると、P2P送金/チップの審査基準で精査される。「100%還元」の条件を厳密に満たし続ける運用負荷がある
4. **二重の決済フロー**: 投げ銭用とCF支援用で別の処理が必要になり、開発・運用コストが増える

### 解決策: 統一モデル

```
ユーザーの体験:
  動画を見る → 「応援する」ボタン → 金額を選ぶ → 完了

裏側の構造:
  全ての金銭移動 = クリエイターのプロジェクトへの「支援」
  全てのプロジェクト = 物理的な成果物が存在する
  全ての決済 = 外部決済（PAY.JP）約3%
  Apple/Google手数料 = 0%
  PF手数料 = 全取引から徴収可能（10%）
```

### Apple審査での説明

| 質問 | 回答 |
|------|------|
| このアプリは何？ | 購入型クラウドファンディングプラットフォーム |
| 課金は何に使われる？ | プロジェクトへの支援。物理的な成果物が存在する |
| なぜIAPを使わないのか？ | ガイドライン3.1.3(e): 物理商品・サービスの購入には外部決済を使用 |
| 投げ銭機能はある？ | ない。全てプロジェクト支援 |

Kickstarter/CAMPFIRE/Makuakeと同じ分類。審査リスク最小。

## プロジェクト要件と「物理的成果物」の定義

### Appleが見ているのは「プロジェクトの性質」

Kickstarterの実態:
- $1 tier = "A thank you email"（物理リターンなし）
- $25 tier = 完成品1個
- $100 tier = 限定版 + サイン入り

$1プレッジに物理リターンは付いていないが、IAPは使っていない。**プロジェクト自体が「物理的な何かを作る」ことだから**、全ティアが外部決済で正当化される。

→ Appleは個々の取引単位ではなく、**プラットフォームの性質**として判断している。

### プロジェクト掲載基準

プロジェクトには「物理的な成果物」が存在すること:

| 例 | 物理的成果物 | 判定 |
|----|-------------|------|
| 古民家カフェを作る | カフェ自体 | ✓ |
| アルバムを制作する | CD/レコード/ライブ | ✓ |
| 映画を撮る | 上映会・DVD | ✓ |
| 商品を開発する | 商品そのもの | ✓ |
| イベントを開催する | イベント自体 | ✓ |
| 「僕の活動を応援してください」（成果物なし） | なし | ✗ 不可 |

これは制限ではなく、CFプラットフォームとして当然の品質基準。「何を作るのか」が明確でないプロジェクトはそもそもCFとして成立しない。

### 物理的商品・サービスの分類（Apple IAP適用判定）

#### 明確にIAP不要

| リターン例 | 根拠 |
|-----------|------|
| グッズ・限定商品の発送 | 物理的商品（3.1.3(e)明確） |
| イベント・ワークショップ参加権 | アプリ外で消費されるサービス |
| 店舗割引（例: カフェ50%オフ） | アプリ外消費の飲食サービス。UberEatsと同構造 |
| 工房見学、制作現場訪問 | アプリ外で消費される体験 |
| サイン入り書籍、手書きお礼状 | 物理的商品 |
| 完成品の先行購入権 | 物理的商品の予約（Kickstarterと同構造） |

#### 明確にIAP必要（避けるべきリターン設計）

| リターン例 | 理由 |
|-----------|------|
| 限定動画（アプリ内視聴） | デジタルコンテンツ |
| デジタルバッジ・称号 | アプリ内消費 |
| 限定フィルター・エフェクト | アプリ内機能 |
| オンラインサロン参加権 | デジタルサービス |

#### グレーゾーン（設計次第で回避可能）

| リターン例 | 回避策 |
|-----------|--------|
| Zoomでの個別相談 | 予約システムをWeb経由にすればアプリ外サービス |
| PDF/電子書籍 | メール配信にすればアプリ外消費 |
| 限定動画をWeb配信 | Webリンクで提供すればIAP不要 |

## バッジ・ステータス表示の設計（Apple IAP回避）

### 問題

支援したことがアプリ上でのステータスになる場合、それが「デジタル商品の購入」と見なされるとIAPが必要になる。

### 判定基準: 「買っている」のか「事実を表示している」のか

| 設計 | Apple判定 | 理由 |
|------|----------|------|
| ¥5,000支援で「ゴールドバッジ」が付く | **NG（IAP必要）** | 金額と紐付いたデジタル商品の購入 |
| ¥10,000以上でプロフィール枠の色が変わる | **NG（IAP必要）** | 金額連動の装飾 = デジタル商品 |
| 「このPJを応援済み」マークが表示される | **OK** | 事実の表示。購入物ではない |
| 支援回数に応じて「◯PJ応援中」と表示 | **OK** | アクティビティの記録（eBay評価と同構造） |
| サポーター一覧に名前が載る | **OK** | Kickstarterの"Backed"表示と同構造 |

### 安全な設計

```
OK（事実の表示）:
  プロフィール: 「12プロジェクト応援中」
  プロジェクトページ: サポーター一覧に名前が載る
  タイムライン: 「◯◯さんがこのプロジェクトを応援しました」
  → 情報表示であり、デジタル商品の販売ではない
  → GitHub Contributions、eBay評価、Yelp Eliteと同構造

NG（デジタル商品の販売）:
  金額に応じたランク別バッジ（ブロンズ/シルバー/ゴールド）
  支援額に応じてプロフィールの枠の色が変わる
  特定金額以上で限定アイコン/エフェクトがアンロック
  → 金額とデジタル特典が直接紐付いている
  → Apple: デジタル商品の販売 → IAP必須
```

### 結論

アプリ内のステータス表示は「支援した事実の反映」として設計すれば安全。金額ランクに連動する装飾・バッジは避ける。

## デジタルリターンの配信ルール

### 「物理リターン1つ + デジタルコンテンツ」はアウトか？

**アプリ内で消費されるデジタルコンテンツの場合はアウト。**

物理リターンを「隠れ蓑」にしてアプリ内デジタルコンテンツを配信する設計は、Appleに「実質デジタルコンテンツの販売」と判断される。

```
アウトな例:
  ¥1,000 → ステッカー1枚（物理）+ アプリ内限定動画
  → Apple: 「ステッカーはおまけ。実質は限定動画を買っている」→ IAP必要

セーフな例:
  ¥1,000 → ステッカー1枚（物理）
  ¥5,000 → Tシャツ + 制作現場見学（物理体験）
  ※ デジタルコンテンツなし or アプリ外で配信
```

### Kickstarterはなぜデジタルリターンでも通っている？

1. **Kickstarterアプリ内でデジタルコンテンツを消費しない** — ゲームキーはSteamで使う、PDFはメールで届く
2. **プロジェクトの主目的が物理的な創作** — デジタルは副次的
3. **Kickstarterはコンテンツ配信プラットフォームではない** — 資金調達の場

### クリエイター向けリターン設計ガイドライン

```
リターン設計のルール:

  ✓ 物理的リターン → 自由に設定可能
    例: 商品、グッズ、体験チケット、イベント参加、店舗割引

  ✓ デジタルコンテンツ（アプリ外配信）→ OK
    例: 限定動画をYouTube限定公開リンクで配信
         PDFをメールで送付
         Zoomミーティングの招待URL

  ✗ デジタルコンテンツ（アプリ内消費）→ NG
    例: アプリ内でしか見れない限定動画
         アプリ内の特別なバッジ・装飾
         アプリ内の特別な機能アンロック

  判断基準:
  「アプリの外で届ける・消費するもの」→ OK
  「アプリの中でしか使えないもの」→ NG
```

このルールを守れば、Kickstarterと完全に同じ構造となり、Apple審査上の判定は白（グレーではない）。

## ティア構造の設計

### 基本構造

```
全プロジェクトに自動生成されるティア:

  ¥100〜    「応援」
             → リターン: プロジェクトページにサポーター名表示
             → 郵送なし、住所不要
             → プロジェクト自体が物理的成果を持つのでIAP不要

  ¥1,000〜  「サポーター」（クリエイターがカスタム設定）
             → リターン例: ステッカー、割引券、イベント参加権
             → 物理リターンの場合のみ住所入力

  ¥5,000〜  「スポンサー」（クリエイターがカスタム設定）
             → リターン例: 完成品、限定グッズ、体験
             → 物理リターンの場合は住所入力
```

### 住所入力のタイミング

```
¥300で応援 → 金額選択 → 決済 → 完了（住所不要）
¥5,000で応援（Tシャツ選択）→ 金額選択 → 決済 → 住所入力 → 完了
```

**小額応援に住所入力を要求しない。** 物理リターン選択者のみ。これによりTikTok的な軽い体験を維持。

### スケーラビリティ

全員に郵送は非現実的（1万人×¥84 = ¥84万の郵送費）。ティア構造で解決:
- 小額ティア: 名前掲載等（デジタル、コストゼロ）
- 中〜大額ティア: 物理リターン（対象者は限定的）

### 早期支援インセンティブ

| インセンティブ | 実装方法 | 安全性 |
|--------------|---------|--------|
| 早期割引（先着N名で20%オフ） | CF支援のバリエーション | ✓ 安全（Makuakeの超早割と同構造） |
| 限定カラー・限定版 | 物理リターンのバリエーション | ✓ 安全 |
| 次回プロジェクト先行アクセス | 通知の優先配信 | ✓ 安全（販売ではない） |
| 「初期支援者」表示 | 支援した事実の表示 | ✓ 安全（金額連動の装飾でなく、事実表示として設計すればOK） |

## プロジェクトなしのクリエイターへの対応

### 方針: 応援はプロジェクト単位のみ

```
プロジェクトがないクリエイター:
  → 「応援する」ボタンは非表示
  → 「フォロー」「通知ON」のみ
  → プロジェクト立ち上げ時に通知が届く
```

### 理由

1. 「このアプリはCFプラットフォームです」という説明との整合性が完璧
2. クリエイターに「プロジェクトを立ち上げないとお金が入らない」という動機づけ
3. 質の高いプロジェクト作りへのインセンティブ
4. 「常設プロジェクト」を許可するとAll or Nothingの原則と矛盾する

## 収益モデルの持続可能性分析

### 統一モデルでの収益試算

```
月間GMV ¥5,000万（初期目標）
  全取引がCF支援 × PF手数料10% = ¥500万/月

  ※ 旧モデル（投げ銭100%還元）の場合:
    CF ¥3,000万 × 10% + 投げ銭 ¥2,000万 × 0% = ¥300万/月
  → 統一モデルは同じGMVで収益が67%増加
```

### 段階別の収益構造

| 収益源 | 初期 | 成長期 | スケール期 |
|--------|------|--------|-----------|
| CF手数料（10%） | **主力** | **主力** | 主力 |
| プロモーション掲載 | - | 開始 | 主要 |
| プレミアムクリエイターツール | - | 開始 | 補助 |
| ブランドコラボ | - | - | 補助 |

### 損益分岐の目安

```
月間コスト（概算）:
  サーバー/CDN: ¥80〜150万
  チーム（3〜5人）: ¥200〜400万
  決済手数料: ¥40万
  その他: ¥50万
  合計: ¥370〜640万

損益分岐GMV:
  ¥370万 ÷ 10% = ¥3,700万/月（楽観）
  ¥640万 ÷ 10% = ¥6,400万/月（悲観）

  → 月間GMV ¥4,000〜6,500万で損益分岐
  → 年間GMV ¥5〜8億
```

### 競合との手数料比較（統一モデル）

| プラットフォーム | 合計手数料 | iOS追加コスト | クリエイター受取率 |
|-----------------|-----------|--------------|-------------------|
| **本PF（想定）** | **約13%** | **なし** | **約87%** |
| Kickstarter | 8〜10% | なし | 約90% |
| CAMPFIRE | 17%+税（約18.7%） | なし | 約81% |
| Patreon | 12〜15% | iOS経由+30% | 55〜88% |

CAMPFIREより安く、Kickstarterに近い水準。動画での発見性という付加価値を考えると競争力あり。

## 対象取引類型（改訂版）

### 旧定義
- 購入型クラウドファンディング（All or Nothing）
- 投げ銭（都度課金）

### 新定義
- **プロジェクト応援**（All or Nothing）: 唯一の取引類型
  - 金額に応じたティアが自動適用される
  - 全プロジェクトに物理的成果物が存在
  - ユーザーは「投げ銭か支援か」を意識しない
  - 小額（¥100〜）から大額まで、同一フローで処理

### 投げ銭を廃止する理由
1. UXが単純化される（1つのフロー）
2. Apple IAP論点が消滅する（P2P送金/チップの審査が不要）
3. 全取引からPF手数料を徴収でき、収益性が大幅に向上
4. 開発・運用コストが削減される（決済フローが1つ）
5. 「CFプラットフォーム」としてのブランドが一貫する
